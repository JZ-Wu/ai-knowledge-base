# 从 Evaluate Idea 到 Propose Idea

> 整理自一位一线研究者在 LLM 时代关于"如何提出有价值 idea"的思考笔记（2026.04）。
> 不是指南，也不是可以直接照搬的流程——是一套仍在演化的个人方法论。

## 引言：为什么要从 "提出 idea" 转向 "评估 idea"

许多研究者实现一个 idea 的速度都很快——哪怕在 ChatGPT 之前，两周内打通 pipeline、看到初步结果并不难。但一个长期困扰的问题是：**如何提出一个真正有价值的 idea？**

一个被忽视的真相：**不止大模型提不出好 idea，人类自己提 idea 也很难**。真正的关键不在"提出"，而在**评估**（Evaluate）。评估做好了，就能跳进 Karpathy 描述的 **Auto Loop**——让大模型海量生成 idea，高效过滤，只把精力放在真正有潜力的那部分。

```
     ┌────────────────┐        ┌────────────────┐
     │   Propose      │───────▶│   Evaluate     │
     │   (海量生成)    │        │   (筛选潜力)    │
     └────────────────┘        └────────┬───────┘
            ▲                           │
            │                           │
            └───────────────────────────┘
                   只在值得的 idea 上
                    投入真实实验资源
```

**核心思路转变**：把评估从"可有可无的直觉判断"升级为"科研工作流的核心齿轮"。

---

## 一、如何评估一个 Idea

### 1.1 人类常见的两种评估方式

| 方式 | 做法 | 本质 | 局限 |
|------|------|------|------|
| ① 丢给 ChatGPT / Claude | 问模型可行性和风险 | **大模型之间在对话**，你只是协调者 | 不显著优于 AutoResearch |
| ② 找同学 / 导师 / 同行讨论 | 多角度反馈 | 不同人带来不同 context | 人类评估方差极大 |

第二种明显比第一种有价值——但依然存在严重的方差问题。**单凭 idea 本身去精准评估它本来就非常困难**。

### 1.2 关键观念：我们不需要完美评估

> 好消息是：**我们并不需要完美评估。**

我们只需要把**明显糟糕的 idea 筛掉**就够了。甚至丢掉一些其实还不错的 idea 也没关系——因为：

- **Idea 几乎是无限的**：丢掉几个并不会阻碍方向探索
- **目标足够 personal**：发出一篇有价值的 paper，而不是穷尽一切可能
- **假阴性成本远低于假阳性成本**：押错方向浪费数月；错过一个点子没什么损失

**启发**：评估器不用完美，只需稳定地过滤明显的噪音。

## 二、如何提出更有价值的 Idea：核心是 Context

大模型（以及人类）提出的 idea 常被批评"缺乏 novelty"。一个核心原因是：**上下文（context）不足**。

### 2.1 类比：导师指导学生

这个过程可以类比导师指导学生的经典场景：

- 导师直接给你几篇核心 paper → 你主动给大模型喂高质量上下文
- 学生自己用 keywords 搜文献 → 大模型靠自己的知识和搜索

刚进一个领域时，最容易做的错事：**只用几个 keywords 搜最相关的几篇 paper**，然后基于这些极其有限的信息 brainstorm idea。这样产生的 idea 自然缺 novelty 和可行性。

更深一层的问题：**知名 paper 的真正价值在于文章细节中作者遇到困难时 decision maker 的过程**——这个内容大模型很难直接 get 到。

### 2.2 有经验研究者的两种 Context 优势

有经验的研究者能提出更有意思的 idea，是因为拥有更丰富的 context：

1. **对当前领域如臂使指**，掌握大量信息
2. **同时了解交叉领域、相关领域**，甚至表面上不相关的领域

看到一个新问题时，他们能自然地把过去在强化学习、计算机视觉、NLP 或其他领域的经验"移植"过来。**这就是 context 的优势**。

### 2.3 提升大模型 idea 质量的三条路径

提升大模型提出 idea 质量的关键，在于**提升它可用的上下文质量**。目前主要有三条路径：

```
路径 ①: 优化模型主动搜索的上下文
  ├── 方式: 换 Gemini 搜索、写更好的 prompt
  ├── 类比: 导师遇到一个"六边形学生"——能自己找高质量 paper、
  │        自带跨领域知识、搜索能力强、代码能力极强
  ├── 效果: 能产生让导师觉得不错的 idea
  └── 判断: 仅切换更强搜索的增益有限，需要配合其他手段

路径 ②: Human in the loop (最立竿见影)
  ├── 方式: 你作为领域内有经验的人，主动给模型注入
  │        跨领域知识、引导思考方向
  ├── 优势: 直接把高价值的 context 注入进去
  └── 判断: 当前阶段性价比最高的方式

路径 ③: 通过真实实验产生的信号迭代
  ├── 核心: idea 不是想出来的, 是从实验中长出来的
  └── 详见下一节 Saining 启发
```

## 三、关键启发：Idea 应该从实验中"长出来"

整理自一位一线研究者对 Saining Xie 长播客的复盘，几个观点彻底改变了对"提出 idea"的认知，并完美印证了"通过实验迭代积累上下文"的思路。

### 3.1 Kaiming 对 Baseline 的极致重视

Kaiming 做顶级工作的一个重要秘诀：

> 在新领域开始时，**亲自把 baseline 复现到极致**，然后在纯技术层面寻找优化空间。

这个过程得到的结果甚至可以接近或超过 SOTA。一个典型例子：**ResNet strikes back: An improved training procedure in timm**——只是对训练配方的精细打磨，就能把 ResNet 性能推到惊人水平。

**启示**：不要轻视"把 baseline 做到极致"这件看似无聊的工作。它本身就是一个强大的信号源。

### 3.2 "三个月后直接 work 的 idea" 意味着什么

> 如果你想出一个 idea，三个月后直接 work，并且投稿的就是这个 idea，那两种可能：
> 一种是你比所有人都聪明很多——**这个基本不可能**；
> 或者第二种——**这不是一个 idea**。

**推论**：任何真正有价值的 idea，都不会一次性、完美地在三个月内验证通过。反过来，"顺利"本身就是个警报信号。

### 3.3 科研本质是 Explore

科研工作流的真实形态：

```
Step 1: 把一个 baseline 或要 follow 的论文 clone 下来
Step 2: 简单复现 paper, 做一些你认为有价值的分析性实验
        → 你会看到一些成功的 signal, 但更多的是失败
Step 3: 从成功/失败中学习
        → 失败往往带来更大的 gradient, 让你学到更多

"如果一个小改动让结果掉了 10 个点，
 反过来想，我们也许就能提升 10 个点。"
```

**核心信念**：失败信号比成功信号更宝贵。失败揭示了约束，成功可能只是运气。

### 3.4 彻底放弃"提出一个完美 idea"

之前让大模型 propose idea 的范式其实带着很强的先验：

```
错误范式: 一次性想出一个 idea → 通过实验证明它确实 work
正确范式: 从当前任务的真实困难和痛点中, idea 自然生长出来
```

这种一次性"灵光一闪"是反常识的，也没有多大价值。**好的 idea 从痛点生长出来**——而要发现这些痛点，需要：

- 大量的领域知识
- 真实的实验信号
- 充分的失败经历

直接指望"灵光一闪"既不现实，也不正常。

### 3.5 Human in the loop（再次出现）

大模型虽然有能力实现所有分析方式，但他**不一定会主动去做**。需要你有一些 insight 告诉他：

- 我们可以尝试用什么手段分析
- 或者简单粗暴地用一个 skill 枚举所有分析方式

**模型提供执行力，人提供方向感**——至少在当前阶段是这样。

### 3.6 一个实践 pipeline

作者实际 follow 的流程：

```
1. 要求当前 orchestra 把实验整理成结构化文档
2. 把文档丢给 GPT-Pro
3. 让 GPT-Pro 分析
   → exciting 的发现: GPT-Pro 会自然地设计更多分析性实验,
     并明确告诉你"我需要这些实验才能做出更精准的判断"
4. 通过 pro2api 让模型主动调用，多轮自动化迭代
```

**一个具体 case**：follow CVPR2026 的一篇 paper，通过这种交互方式，在其两个数据集上分别提升了 **6 个点**和 **10 个点**。

## 四、方法论提炼

```
┌────────────────────────────────────────────────────────┐
│  科研不是灵光一闪的艺术, 而是一场漫长的、迭代的探索。         │
│                                                        │
│  好的 idea 不是被"想出来"的, 而是从大量真实实验产生的       │
│  信号中, 一点点"长出来"的。                                │
└────────────────────────────────────────────────────────┘
```

### 四个可操作的原则

1. **评估 >> 生成**：与其纠结想出好 idea，不如先建立稳定的筛选机制。评估只要能过滤掉明显糟糕的就够了。

2. **Context 决定质量**：idea 的 novelty 上限由可用上下文的丰富度决定。投资 context（深度阅读、跨领域阅读、Human in the loop）比投资"灵感训练"更值得。

3. **Baseline 是 gradient 的最大来源**：把 baseline 复现到极致，是新领域入场的最高性价比动作。一个掉 10 分的小改动，反过来可能就是涨 10 分的门。

4. **承认 idea 长周期成形**：真正有价值的 idea 不会三个月内顺利兑现。顺利 = 警报。把"提出 idea"视为一个**与实验协同进化**的慢过程。

### 与 LLM 协作的工作模式雏形

```
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  Human   │◀──▶│   LLM    │◀──▶│  Experi- │
  │ (方向感) │    │ (执行力) │    │   ment   │
  └──────────┘    └──────────┘    └──────────┘
       ▲              ▲               │
       │              │               │
       └──── 信号 ────┴───── 反馈 ────┘

核心: 三方闭环。任何一方缺失都会退化为
     AutoResearch / 纯手工科研 / 盲目跑实验。
```

## 五、开放问题

作者提出的、仍在探索中的核心方向：

> 如何让模型真正获取到更高层次的信息来源，并像一位非常 senior 的 scientist 那样进行思考？

这是当前 LLM 辅助科研的主要瓶颈：
- 模型能做执行、搜索、总结
- 但缺乏**经验带来的直觉性优先级判断**
- 缺乏对**问题重要性的品味**

这部分能否通过更好的 context、更长的记忆、更结构化的实验反馈来逐步注入，是接下来几年值得深挖的方向。
